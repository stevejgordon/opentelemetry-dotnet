# OpenTelemetry .NET Diagnostic: OTEL1007

## Overview

This is an Experimental API diagnostic covering the following APIs:

* `IDeclarativeConfigProperties`
* `ComponentFactory<TComponent>`
* `SamplerFactory`
* `SpanProcessorFactory`
* `SpanExporterFactory`
* `ComponentRegistrationExtensions.AddSamplerFactory`
* `ComponentRegistrationExtensions.AddSpanProcessorFactory`
* `ComponentRegistrationExtensions.AddSpanExporterFactory`

Experimental APIs may be changed or removed in the future.

## Details

The OpenTelemetry Specification defines a
[declarative configuration](https://opentelemetry.io/docs/languages/sdk-configuration/declarative-configuration/)
model that allows SDK components (samplers, processors, exporters) to be
instantiated from a YAML file. The APIs covered by this diagnostic provide the
factory engine through which built-in and third-party components are registered
and dispatched.

### Factory model

Each factory is identified by a `(ComponentType, Name)` pair, where `Name`
matches the schema-level key used in the YAML file (e.g. `always_on`,
`trace_id_ratio_based`, `simple`, `console`). The
`ComponentProviderRegistry` (internal) is built once from all registered
factories at provider-construction time and dispatched by the declarative
walker and the env-var compatibility adapter.

### Registering a custom factory

```csharp
// 1. Implement a factory.
[Experimental("OTEL1007")]
public sealed class AcmeSamplerFactory : SamplerFactory
{
    public override string Name => "acme_sampler";

    public override Sampler CreateComponent(
        IDeclarativeConfigProperties properties,
        IServiceProvider serviceProvider)
    {
        properties.TryGetDouble("ratio", out var ratio);
        return new AcmeSampler(ratio ?? 1.0);
    }
}

// 2. Register the factory before building the provider.
services.AddSamplerFactory(new AcmeSamplerFactory());

// 3. Reference the sampler in the YAML file.
// tracer_provider:
//   sampler:
//     acme_sampler:
//       ratio: 0.5
```

### Property bag (`IDeclarativeConfigProperties`)

The property bag preserves the three-state distinction from the YAML source:

| State | `TryGet*` return | `out value` | `GetKeys()` |
|-------|-----------------|-------------|-------------|
| Absent | `false` | - | excluded |
| Present-null | `true` | `null` | included |
| Present | `true` | the value | included |

This distinction is required by the specification (`nullBehavior` at `Create`
time differs from absent).

### Reloadable values

Structural configuration (which component to create, nested children) comes
from the property bag. Reloadable scalar values (e.g. a sample rate that
changes at runtime) should be read from
`IOptionsMonitor<TOptions>` resolved from the `IServiceProvider` passed to
`CreateComponent`. Only value reload is supported; structural changes (e.g.
switching sampler type at runtime) require provider rebuild.
